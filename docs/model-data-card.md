# Model/Data Card

Короткая карточка фиксирует, на каких данных обучена модель, что именно она
предсказывает и где границы применимости прототипа.

## Назначение

Проект использует набор UCI Bank Marketing как компактный и понятный пример
прикладной ML-задачи: по данным клиента и контакта маркетинговой кампании
оценить вероятность положительного ответа на банковское предложение. Набор
удобен для мониторинга, потому что содержит числовые и категориальные
признаки, дисбаланс классов, вероятностную модель, отложенную целевую метку и
реалистичные источники дрейфа распределений.

## Target

Целевая переменная исходного датасета `y` преобразуется в бинарный `target`:

- `1`: клиент подписал срочный депозит, исходное значение `yes`;
- `0`: клиент не подписал срочный депозит, исходное значение `no`.

Это бинарная классификация, потому что операционное решение в задаче имеет
два исхода: считать заявку/контакт положительным или отрицательным. Модель
при этом возвращает вероятность класса `1`, а решение принимается по
operating threshold, выбранному на validation split по F1.

## Данные и разбиение

Используется полный Bank Marketing dataset после очистки target и удаления
leakage-признака `duration`.

| split | rows | positive rows | positive rate |
| --- | ---: | ---: | ---: |
| train | 27 126 | 1 367 | 5.04% |
| validation | 9 042 | 1 066 | 11.79% |
| test | 9 043 | 2 856 | 31.58% |
| all | 45 211 | 5 289 | 11.70% |

Разбиение детерминированное и сохраняет исходный порядок строк: первые 60%
используются для обучения, следующие 20% для калибровки и выбора threshold,
последние 20% для тестовой оценки. Это полезно для демонстрации мониторинга:
между частями данных заметен сдвиг base rate, поэтому система должна уметь
обнаруживать дрейф и деградацию качества. Для продуктовой модели такое
разбиение нужно дополнительно валидировать по реальной временной оси кампаний.

## Признаки

Числовые признаки:

- `age`
- `balance`
- `day_of_week`
- `campaign`
- `pdays`
- `previous`

Категориальные признаки:

- `job`
- `marital`
- `education`
- `default`
- `housing`
- `loan`
- `contact`
- `month`
- `poutcome`

Категориальные признаки кодируются через `OneHotEncoder(handle_unknown="ignore")`.
Числовые признаки проходят median imputation и standard scaling. Категории
без значения в исходных данных нормализуются как строковые значения при
подготовке профиля и обработке признаков.

## Почему удален `duration`

`duration` отражает длительность последнего контакта с клиентом. Этот признак
становится известен только после завершения контакта, поэтому до принятия
решения он недоступен. Если оставить его в обучении, модель получит утечку
постфактум-информации: offline-метрики станут лучше, но serving-сценарий будет
нереалистичным. Поэтому `duration` удаляется в training pipeline и записан в
`baseline_profile.json` как `dropped_columns`.

## Модель

Базовая модель проекта:

- `ColumnTransformer` для числовых и категориальных признаков;
- `LogisticRegression(class_weight="balanced")`;
- `CalibratedClassifierCV(method="sigmoid")` на validation split;
- threshold выбирается по максимуму F1 на validation split.

Такой выбор намеренно простой: задача проекта не построить максимальную
скоринговую модель, а показать воспроизводимый контур serving, drift/quality
monitoring, delayed labels, actionable alerts и rollback действий.

## Дисбаланс классов

Положительный класс редкий: во всем наборе около 11.70% положительных
примеров, а в training split только 5.04%. Поэтому:

- модель использует `class_weight="balanced"`;
- основными метриками являются не только ROC AUC, но и PR AUC, precision,
  recall, F1, calibration metrics и доля predicted positive;
- monitoring отслеживает `positive_rate_true`, `positive_rate_pred`,
  `score_psi`, calibration и proxy-сигналы слепого периода.

## Ограничения применимости

- Это учебный прототип для магистерской работы, а не production scoring
  system для реального банка.
- Данные описывают исторические маркетинговые кампании; их нельзя без
  проверки переносить на другой банк, регион, продукт или канал коммуникации.
- Сильный сдвиг target rate между split-ами полезен для демонстрации
  мониторинга, но требует осторожной интерпретации offline-метрик.
- Модель не доказывает причинную связь между признаками и подпиской; она
  оценивает статистическую вероятность в рамках наблюдаемых данных.
- В проекте нет полноценного fairness-аудита по демографическим группам.
- Решения о реальных клиентах должны учитывать business cost, правовые
  требования, согласия на обработку данных и ручной контроль спорных случаев.
