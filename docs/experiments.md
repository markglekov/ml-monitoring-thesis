# Experiment Scenarios

This document fixes a reproducible set of monitoring experiments for the
project demo. All commands assume the Docker stack is already running.

## Preconditions

1. Model artifacts exist under `artifacts/models/` and
   `artifacts/baselines/`.
2. The stack is running:

```bash
make docker-up
```

3. Commands are executed from the repository root.

## Drift Scenarios

The simulator writes manifests to `artifacts/reports/manifests/`. Use a
stable `--seed` so the scenarios are reproducible.

### Baseline-like Scenario

```bash
UV_CACHE_DIR=.uv-cache uv run python -m app.simulator.generate_stream \
  --source-split train \
  --rows 300 \
  --scenario none \
  --segment-key drift_none_demo \
  --seed 42

UV_CACHE_DIR=.uv-cache uv run python -m app.monitoring.drift_job \
  --window-size 300 \
  --segment-key drift_none_demo \
  --min-rows 50
```

Expected outcome:

- `overall_drift` should usually stay `false`
- false positives should be limited

### Mild Drift Scenario

```bash
UV_CACHE_DIR=.uv-cache uv run python -m app.simulator.generate_stream \
  --source-split train \
  --rows 300 \
  --scenario mild \
  --segment-key drift_mild_demo \
  --seed 42

UV_CACHE_DIR=.uv-cache uv run python -m app.monitoring.drift_job \
  --window-size 300 \
  --segment-key drift_mild_demo \
  --min-rows 50
```

Expected outcome:

- several drifted features appear
- `overall_drift` should usually become `true`

### Severe Drift Scenario

```bash
UV_CACHE_DIR=.uv-cache uv run python -m app.simulator.generate_stream \
  --source-split train \
  --rows 300 \
  --scenario severe \
  --segment-key drift_severe_demo \
  --seed 42

UV_CACHE_DIR=.uv-cache uv run python -m app.monitoring.drift_job \
  --window-size 300 \
  --segment-key drift_severe_demo \
  --min-rows 50
```

Expected outcome:

- drift becomes obvious across more features than in the mild case
- score distribution drift should usually be visible as well

### Multivariate Drift Scenario

Generate a severe segment and inspect the `__multivariate__` detector in
`drift_metrics` or `GET /monitoring/drift/runs`.

Expected outcome:

- the domain-classifier detector should report `warning` or `critical`
- the run summary should include `multivariate_drift` details

## Quality Scenarios

Quality monitoring needs delayed labels. Reuse the manifests generated by
the simulator and vary only the label policy.

### Baseline-like Quality

```bash
UV_CACHE_DIR=.uv-cache uv run python -m app.simulator.generate_stream \
  --rows 1000 \
  --scenario none \
  --segment-key quality_base_demo \
  --seed 42

UV_CACHE_DIR=.uv-cache uv run python -m app.monitoring.backfill_labels \
  --label-policy perfect

UV_CACHE_DIR=.uv-cache uv run python -m app.monitoring.quality_job \
  --window-size 1000 \
  --segment-key quality_base_demo \
  --baseline-source test
```

Expected outcome:

- `degraded_metrics_count` should stay close to zero

### Mild Quality Degradation

```bash
UV_CACHE_DIR=.uv-cache uv run python -m app.simulator.generate_stream \
  --rows 1000 \
  --scenario none \
  --segment-key quality_mild_demo \
  --seed 42

UV_CACHE_DIR=.uv-cache uv run python -m app.monitoring.backfill_labels \
  --label-policy custom_flip \
  --flip-prob 0.10

UV_CACHE_DIR=.uv-cache uv run python -m app.monitoring.quality_job \
  --window-size 1000 \
  --segment-key quality_mild_demo \
  --baseline-source test
```

Expected outcome:

- some degraded metrics should appear
- `roc_auc`, `recall`, `f1`, or `brier_score` may move first

### Severe Quality Degradation

```bash
UV_CACHE_DIR=.uv-cache uv run python -m app.simulator.generate_stream \
  --rows 1000 \
  --scenario none \
  --segment-key quality_severe_demo \
  --seed 42

UV_CACHE_DIR=.uv-cache uv run python -m app.monitoring.backfill_labels \
  --label-policy custom_flip \
  --flip-prob 0.25

UV_CACHE_DIR=.uv-cache uv run python -m app.monitoring.quality_job \
  --window-size 1000 \
  --segment-key quality_severe_demo \
  --baseline-source test
```

Expected outcome:

- `degraded_metrics_count` should be materially higher than in the mild case
- severe degradation should be visible in Grafana and the overview API

### Blind-Period Proxy Monitoring

Run only the inference stream and quality job without backfilling labels.

```bash
UV_CACHE_DIR=.uv-cache uv run python -m app.simulator.generate_stream \
  --rows 1000 \
  --scenario severe \
  --segment-key quality_proxy_demo \
  --seed 42

UV_CACHE_DIR=.uv-cache uv run python -m app.monitoring.quality_job \
  --window-size 1000 \
  --segment-key quality_proxy_demo \
  --baseline-source test
```

Expected outcome:

- the quality job should finish with `status=completed_proxy`
- proxy metrics such as `score_psi`, `near_threshold_rate`, or `score_entropy`
  may degrade before labels are available
- an open incident may appear in `GET /monitoring/incidents`

### Segment-Localized Monitoring

Set `MONITORING_SEGMENTS=drift_none_demo,drift_severe_demo` for the scheduler or
run jobs manually with `--segment-key`.

Expected outcome:

- only the affected segment should open drift or quality incidents
- the overview should show segment-local activity and active incidents

## Where Results Are Saved

- Raw prediction events: `inference_log`
- Delayed labels: `ground_truth`
- Drift runs: `monitoring_runs`, `drift_metrics`
- Quality runs: `quality_runs`, `quality_metrics`
- Monitoring incidents: `monitoring_incidents`
- Simulator manifests: `artifacts/reports/manifests/`

## How To Inspect Results

Overview and monitoring APIs:

- `GET /monitoring/overview`
- `GET /monitoring/drift/runs?limit=10`
- `GET /monitoring/quality/runs?limit=10`
- `GET /monitoring/incidents?limit=10`

Prometheus / Grafana:

- `http://localhost:9090`
- `http://localhost:3000`
- `http://localhost:3000/d/ml-monitoring-overview`
- `http://localhost:3000/d/ml-monitoring-drift`
- `http://localhost:3000/d/ml-monitoring-quality`
- `http://localhost:3000/d/ml-monitoring-proxy`
- `http://localhost:3000/d/ml-monitoring-segments`

PostgreSQL:

```bash
make db-shell
```

Useful queries:

```sql
SELECT id, status, segment_key, overall_drift, drifted_features_count
FROM monitoring_runs
ORDER BY id DESC
LIMIT 10;

SELECT id, status, segment_key, degraded_metrics_count, labeled_rows
FROM quality_runs
ORDER BY id DESC
LIMIT 10;
```

## Defense Demo Flow

Use the dashboards in this order during the defense:

1. Start with the executive overview to show current severity, active incidents,
   freshness of jobs, and whether the system is healthy at a glance.
2. Open the drift dashboard and compare baseline-like, mild, and severe drift.
   Show the segment table and the `__multivariate__` detector row.
3. Open the quality dashboard to explain labeled degradation, threshold effects,
   and calibration changes through `roc_auc`, `f1`, `brier_score`, and `ece`.
4. Open the proxy dashboard to demonstrate the blind period where labels are not
   available yet but `score_psi`, `near_threshold_rate`, or `score_entropy`
   already react.
5. Finish with the segments and incidents dashboard to show that the system can
   localize issues to one segment and preserve recommended actions over time.
